# 근거 기반 전문가 패널 v3.1

$ARGUMENTS

## 6축 매핑 (Harness Engineering)

| 축 | 단계에서 적용 | 메커니즘 |
|----|--------------|----------|
| **구조** | 전 단계 | 7단계 layered phase + ✓ Gate + 미달 시 fallback |
| **맥락** | 3단계 | `AskUserQuestion`으로 사용자 상황 수집 |
| **계획** | 1-2단계 | 학문 지도 → 근거 지형 (탐색 전 설계) |
| **실행** | 4-6단계 | 근거 태그 강제, Core/Peripheral 분리, MVE 설계 |
| **검증** | 게이트 + 종료 체크리스트 | ✓ Gate + Checklist Before Stopping |
| **개선** | 7단계 + PBE Loop | 옵시디언 저장 → 후속 회고에서 N-of-1 축적 |

## 실행 단계

### ▶ [1/7] 학문 지도 그리기 (Science Mapping)

`$ARGUMENTS`에서 주제를 파악한 후, **관련 학문 분야를 먼저 식별**한다.

수행 항목:
- 주제와 관련된 학문 분야(XX Science, XX Psychology, XX Studies 등) **2-4개** 식별
- 각 분야에서 이 주제에 대해 어떤 종류의 연구가 축적되어 있는지 개요 제시
- 가장 관련성 높은 1-2개 분야를 **주 근거원(primary evidence source)**으로 선택

출력 형식:
```markdown
## 학문 지도

| 분야 | 관련성 | 핵심 연구 주제 | 근거 풍부도 |
|-----|--------|---------------|-----------|
| [분야명] | 높음/중간 | [이 분야에서 다루는 관련 주제] | 풍부/보통/희소 |

**주 근거원**: [선택한 1-2개 분야와 선택 이유]
```

> ✓ Gate: 학문 분야 2개 이상 식별 + 주 근거원 선택 완료. 미충족 시 → 인접 학문 분야로 확대 탐색.

---

### ▶ [2/7] 근거 지형 탐색 (Evidence Landscape)

주 근거원 분야에서 **핵심 발견들을 정리**한다. **WebSearch를 사용하여** 메타분석, 체계적 리뷰를 우선 탐색한다.

수행 항목:
- **WebSearch**로 해당 분야의 메타분석, 체계적 리뷰, 주요 연구를 탐색 (최근 10년 이내 우선)
- WebSearch 사용 불가 시 내부 지식으로 진행하되, 모든 발견에 `[Internal Knowledge]` 태그 부착
- **설명적 근거 (Descriptive)**: 이 현상이 어떻게 작동하는가? 무엇이 관찰되는가?
- **처방적 근거 (Prescriptive)**: 어떻게 하면 더 효과적인가? 무엇이 검증되었는가?
- 각 발견에 **근거 수준** 태그 부여

근거 수준 체계 (`references/evidence-framework.md` 참조):
- **[Strong]**: 메타분석, 체계적 리뷰, 복수의 재현된 실험
- **[Moderate]**: 복수의 RCT 또는 대규모 관찰 연구
- **[Emerging]**: 단일 연구, 예비 결과, 파일럿 스터디
- **[Expert]**: 전문가 합의, 임상 경험 기반 (근거 약함 — 주의 필요)
- **[Contested]**: 상반된 연구 결과가 존재

출력 형식:
```markdown
## 근거 지형

### 설명적 발견 (이렇게 작동한다)
- [Strong] 발견 내용 — 근거 출처 (저자, 연도)
- [Moderate] 발견 내용 — 근거 출처

### 처방적 발견 (이렇게 하면 효과적이다)
- [Strong] 발견 내용 — 근거 출처
- [Emerging] 발견 내용 — 근거 출처

### Gottman 경고 (근거 없이 퍼진 통념)
- [Contested] 흔히 믿지만 근거가 약하거나 반증된 것들
```

> **Gottman 경고**: 김창준의 Gottman 사례에서 따온 이름. "친구 사귀기 전문가"의 프로토콜이 실제로는 역효과를 냈듯, 근거 없이 전문가 권위만으로 퍼진 통념을 명시적으로 플래그한다.

> ✓ Gate: 설명적 + 처방적 발견 각 2개 이상 + 근거 수준 태그 부여 완료. 미충족 시 → 주제를 좁히거나 인접 분야 근거로 보충.

### ▶ [3/7] 컨텍스트 수집 (User Gate)

사용자의 **구체적 상황과 맥락**을 파악한다. **반드시 `AskUserQuestion` 도구로 수집**한다.

수행 항목:
1. 이전 대화 맥락에서 관련 정보 추출
2. 프로젝트의 CLAUDE.md에서 사용자 컨텍스트 참조
3. 누락된 핵심 맥락(현재 상황·제약·목표) 중 1-3개를 `AskUserQuestion`으로 질문

표준 질문 패턴:
```
AskUserQuestion(
  question: "현재 [주제]와 관련해 어떤 상황에 가장 가까우세요?",
  header: "현재 상황",
  options: [
    { label: "[옵션 A]", description: "구체 시나리오" },
    { label: "[옵션 B]", description: "구체 시나리오" },
    { label: "[옵션 C]", description: "구체 시나리오" }
  ]
)
```

이 단계의 핵심: 4단계 전문가 토론이 **사용자의 실제 맥락에 맞춰** 진행되도록 정보 수집.

> ✓ Gate: 사용자 맥락 파악 완료 (최소: 현재 상황 + 원하는 변화). 미충족 시 → 추가 `AskUserQuestion` 호출.

### ▶ [4/7] 근거 기반 패널 토론

> ⚠️ 이 토론은 Claude가 연구 기록과 학습 데이터를 기반으로 시뮬레이션한 것입니다. 실제 전문가의 현재 견해와 다를 수 있습니다.

**근거 지형 위에서** 전문가 패널을 소환하여 토론한다.

#### 전문가 선정 (6인)

선정 기준 (**권위가 아닌 근거 기여도** 기준):
- 해당 분야에서 핵심 연구를 수행한 연구자 (최소 3인)
- 해당 근거를 실무에 적용한 실천가 (최소 1인)
- 다른 분야에서 교차 관점을 제공할 수 있는 전문가 (최소 1인)
- **Implementation 전문가** 또는 **비판적 관점** 보유자 (최소 1인)

`--experts` 플래그로 사용자 직접 지정 가능 (이 경우에도 근거 기반 토론 원칙은 유지).

#### 토론 구조

1. **전문가 소개**: 이름, 전문 분야, **핵심 연구 기여** (저서보다 연구 기여 우선)
2. **근거 기반 라운드**: 2단계에서 발견한 근거에 대해 각 전문가가 해석/확장/반박
3. **사용자 맥락 적용 라운드**: 이 근거가 사용자의 구체적 상황에서 어떻게 적용되는가
4. **교차 검증**: 전문가 간 의견 충돌 시 **어떤 근거에 기반한 의견인지** 명시하며 토론

#### 토론 규칙

- 모든 주장에 근거 수준 태그 필수: `[Strong]`, `[Moderate]`, `[Emerging]`, `[Expert]`, `[Contested]`
- `[Expert]` 수준의 의견은 반드시 "이것은 근거가 제한적이며, 제 경험에 기반한 의견입니다"를 명시
- 만장일치더라도 근거가 약하면 그 사실을 솔직하게 표기

> ✓ Gate: 전문가 6인 선정 + 4라운드 토론 완료 + 모든 주장에 근거 수준 태그 부여. 미충족 시 → 전문가 수를 3-4인으로 축소하되 교차 분야 1인 필수 포함.

### ▶ [5/7] 구현 설계 (Implementation Design)

Implementation Science 원칙을 적용하여 **실제로 작동하는 실행 계획**을 만든다 (`references/implementation-guide.md` 참조).

> "좋은 베스트프랙티스도 구현 방법이 잘못되면 실패한다" — 빌 게이츠 재단 탁월한 교사 프로젝트 실패 사례

출력 형식:
```markdown
## 구현 설계

### Core Practice (절대 바꾸면 안 되는 핵심)
- 이 원칙/방법의 효과를 만드는 최소한의 핵심 요소들

### Peripheral Practice (맥락에 맞게 변형 가능)
- 핵심 기능은 유지하되 형태는 자유롭게 적응 가능한 요소들

### 적응 설계 (Adaptation)
- 사용자의 구체적 맥락(직업, 환경, 도구, 습관)에 맞춘 구현 방법

### 이해관계자 고려
- 이 변화가 영향을 미치는 사람들

### 퍼실리테이션 포인트
- 혼자 하기 어려운 부분, 활용 가능한 도구/자원
```

> ✓ Gate: Core Practice 최소 2개 + Peripheral Practice 최소 2개 식별. 미충족 시 → 4단계 토론에서 합의된 핵심 원칙 기반으로 재구성.

### ▶ [6/7] 실천 실험 설계 (PBE Loop)

EBP(근거→실천)에서 PBE(실천→근거)로 이어지는 **선순환 구조**를 설계한다.

출력 형식:
```markdown
## 실천 실험

### 최소 실험 (Minimum Viable Experiment)
- 1-2주 안에 시도할 수 있는 가장 작은 실험
- 명확한 성공/실패 판단 기준
- 측정 방법 (주관적 체감 + 가능하면 객관적 지표)

### 관찰 포인트
- 실험하면서 주의 깊게 관찰할 것들

### 근거 축적 방법
- 실험 결과를 어떻게 기록할 것인가
- N회 반복 후 자신만의 Practice-Based Evidence 정리

### 반복 주기
- 언제 실험을 평가하고 조정할 것인가
```

> ✓ Gate: MVE 1개 이상 + 성공/실패 기준 명시 + 반복 주기 설정. 미충족 시 → 5단계 Core Practice 중 가장 작은 것으로 MVE 재설계.

### ▶ [7/7] 저장 (User Gate)

6단계 완료 후 사용자에게 저장 여부를 확인한다.

```
AskUserQuestion(
  question: "전체 토론을 어디에 저장할까요?",
  header: "저장 위치",
  options: [
    { label: "Obsidian (3.Resource/ebp/)", description: "/obsidian-save 호출 — EBP_{주제}.md" },
    { label: "현재 디렉토리 .md 파일", description: "Write 도구로 EBP_{주제}.md 저장" },
    { label: "저장 안 함", description: "대화창 출력만 유지" }
  ]
)
```

**중요: 요약하지 말고 1~6단계 전문을 그대로 저장한다.** 임의 축약 금지.

저장 시 메타데이터:
- 파일명: `EBP_{주제요약}.md`
- 태그: `ebp`, `전문가패널`, `#{관련학문분야1}`, `#{관련학문분야2}`
- 본문 상단: `> 생성일: [[{YYYY-MM-DD}]]`

> ✓ Gate: 사용자 선택대로 저장 완료 + 경로 안내. 미충족 시 → 전체 내용을 코드블록으로 출력하여 수동 복사 가능하게.

`--quick` 플래그 사용 시:

## 톤 & 스타일

- 각 전문가의 실제 어투와 사상을 최대한 반영
- 전문 용어는 영어 OK, 설명은 한국어
- **비판단적이고 호기심 어린 톤** — 근거가 약한 것을 "틀렸다"가 아니라 "아직 검증이 부족하다"로 표현
- 실용적: 토론으로 끝나지 않고 반드시 **구현 가능한 실험**으로 연결
- 정직: 근거가 없으면 없다고 솔직하게 말한다

**Good**:
> **Kahneman**: [Moderate] Stanovich의 연구에 따르면, 시스템 2의 개입은 인지 부하가 높을 때 현저히 감소합니다. 다만 이것은 근거가 제한적이며, 실험실 환경에서의 결과임을 고려해야 합니다.

**Bad**:
> **Kahneman**: 시스템 1과 시스템 2가 있는데요, 시스템 2를 잘 활용하면 더 좋은 의사결정을 할 수 있습니다!

## Hard Rules

1. **근거 수준 태그 필수** — 4단계 모든 주장에 5단계 태그 중 하나 부여
2. **권위 논증 금지** — "X가 말했다"만으로 근거 인정 불가, 근거 출처 별도 명시
3. **AskUserQuestion 사용 강제** — 3단계와 7단계는 반드시 도구 호출로 진행 (텍스트로 묻지 않음)
4. **`[Expert]` 등급은 명시적 경고 동반** — "근거 제한적, 경험 기반" 문구 필수
5. **WebSearch 결과는 출처 인용** — 가능하면 저자·연도·저널 표기

## Edge Cases

| 시나리오 | 기대 동작 |
|---------|----------|
| 비학문적 주제 ("오늘 저녁 뭐 먹을까?") | 가장 가까운 응용 분야 탐색 (예: Food Science, Nutrition) |
| 극도로 좁은 니치 주제 | `[Expert]` 수준 의견이 주가 됨을 사전 고지, 인접 분야 유사 근거 활용 |
| 복수 주제 동시 요청 | 주제를 분리하여 하나씩 처리할 것을 제안 |
| 전문가 6인 구성 불가 (니치 분야) | 3-4인으로 축소, 교차 분야 1인 필수 포함 |
| WebSearch 사용 불가 환경 | 내부 지식으로 진행, 모든 근거에 `[Internal Knowledge]` 태그 |
| 주제가 너무 넓음 ("건강") | 사용자에게 좁히기 질문 ("수면 건강? 심혈관 건강? 정신 건강?") |

---

## Checklist Before Stopping

- [ ] 1단계: 학문 분야 2개 이상 + 주 근거원 선택
- [ ] 2단계: 설명적·처방적 발견 각 2개 이상 + 근거 수준 태그
- [ ] 3단계: `AskUserQuestion`으로 사용자 맥락 수집 완료
- [ ] 4단계: 전문가 6인(또는 축소 구성) 선정 + 4라운드 토론 + 전 주장 태그
- [ ] 5단계: Core Practice 2+ + Peripheral Practice 2+
- [ ] 6단계: MVE 1개 + 성공/실패 기준 + 반복 주기
- [ ] 7단계: `AskUserQuestion`으로 저장 위치 확인 + 사용자 선택대로 처리
- [ ] 모든 주장에 근거 수준 태그 부여
- [ ] `[Expert]` 등급 의견에 경고 문구 동반
